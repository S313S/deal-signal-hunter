# Follow-Up Templates

Use this reference when the user needs a message to send to a client.

## Message Rules

- Start with the relationship, then set the boundary.
- Keep WeChat messages short enough to read on mobile.
- Do not mention private fear, cash pressure, or desperation.
- Do not say "if not suitable I can give it to you" unless the user explicitly chooses to walk away.
- Give one next action, not five.

## Price Optimization Request

```text
我理解你们希望前期费用更轻一点，也愿意把长期合作这个方向一起考虑。

但我建议把两件事分开：当前已经完成/即将交付的基础建设费用，以及后续上线后基于效果的长期合作。前面这部分主要覆盖方案、视觉、页面结构、文案、前端搭建和上线准备，还是需要一个基础费用来承接。

如果预算需要优化，我们可以在价格和服务范围之间做取舍。我可以给你一版更适合前期阶段的方案，但对应交付范围也要一起说清楚。
```

## Revenue Share Instead of Build Fee

```text
分成这个方向我觉得后面可以聊，也感谢你们是往长期合作考虑。

不过我建议不要用后续营业额分成来替代当前网站/工作流的基础建设费用。因为现在还没有正式上线，也还没有稳定营业额，前期这部分主要是已经发生的制作和上线准备成本。

比较稳的方式是：当前阶段先按基础建设费结算；后续如果正式上线并产生可核算订单，我们再单独谈分成比例、统计口径、结算周期，以及我具体负责哪些增长动作。
```

## Client Delays Materials

```text
我这边想把后续推进方式说清楚一点，避免我们一直停在口头沟通里。

如果要继续往正式合作推进，我建议先确认三件事：本阶段交付范围、你们需要补齐的资料和时间、启动费用/后续维护怎么分开算。

资料和启动款确认后，我再给你们排具体交付时间；如果时间间隔比较久，后续排期和报价也需要重新确认。
```

## Test Fee Explanation

```text
测试费主要不是 API 成本，而是技术验证和样片制作成本。

测试阶段我会先拆目标效果、验证工具链、跑一个小样，并记录实际难点和消耗。这样后续正式报价不是拍脑袋，而是基于测试结果来确认。
```

## Soft Close

```text
我建议我们先把这一阶段定下来：交付什么、什么时候交、费用和后续维护怎么分开。

确认后我就按这个范围推进；如果你们现在还没准备好，也可以先暂停，等资料和预算明确后再重新排期。
```
